# ProveGuard

A Flask API that detects whether submitted text was written by a human or generated by AI. It combines an LLM-based semantic signal with stylometric surface analysis, writes every submission to an audit log, and provides a transparency label and an appeal endpoint for disputed classifications.

---

## Setup

**Prerequisites:** Python 3.11+, a [Groq API key](https://console.groq.com/)

```bash
# 1. Clone and enter the project

python -m venv .venv
```

## API endpoints

### `POST /submit`

Classifies a piece of text. Rate-limited to **10 requests per minute / 100 per day** per IP.

```json
// Request
{ "text": "...", "creator_id": "user-123" }

// Response
{
  "content_id": "uuid",
  "attribution": "likely_ai",
  "confidence": 0.81,
  "label": "LIKELY_AI",
  "block": false,
  "banner": "This content may be AI-generated. Review before trusting.",
  "signal_scores": { "llm": 0.85, "stylometric": 0.749 }
}
```

### `POST /appeal`

Disputes a classification. Updates the log row status to `under_review`.

```json
// Request
{ "content_id": "uuid", "creator_reasoning": "I wrote this myself because..." }

// Response
{ "content_id": "uuid", "status": "under_review", "message": "Appeal received and is pending human review." }
```

### `GET /log`

Returns the 10 most recent audit log entries.

## Detection signals

### Signal 1 — LLM semantic analysis (weight: 0.6)

The primary signal calls **llama3-8b-8192 via the Groq API** with a zero-temperature system prompt instructing the model to return a JSON object with an `attribution` field and a `score` in [0.0, 1.0]. The model reasons over the full text semantically — it can pick up on rhetorical structure, claim hedging, vocabulary formality, and the distinctive "helpful AI assistant" register that no surface-level formula captures.

This signal gets the higher weight (0.6) because it reasons about meaning, not just form. A text can have highly variable sentence lengths (human-like surface) but still read unmistakably like AI in its argument structure. The LLM signal catches that; the stylometric signal cannot.

**What I would change for production:** The Groq call adds ~0.5–1.5 s latency and costs money per request. For high-volume use I would batch classify with an embedding-based classifier fine-tuned on labeled examples, reserving the LLM call for borderline cases (0.35–0.65 confidence range) where the extra reasoning power matters.

### Signal 2 — Stylometric surface analysis (weight: 0.4)

Three sub-metrics, each measuring a surface pattern that AI and human writing distribute differently:

**A. Sentence length variance / Coefficient of Variation (weight: 0.5 within Signal 2)**

Human writers produce irregular sentence rhythms — a long tangled clause, then a fragment, then a question, then two short declaratives. AI writers produce consistent sentence lengths because sampling from a language model tends toward modal-length outputs. The coefficient of variation (std dev / mean sentence length) measures this irregularity. High CV → human-like → low AI score.

The denominator is clamped at `min(cv, 1.0)` rather than the spec's original `cv / 2.0`. During calibration, casual human writing (Sample A) hit CV ≈ 0.63, which `cv / 2.0` mapped to 0.68 — deep in the AI-like zone. The denominator of 1.0 places CV = 0.63 at score = 0.37, which is correctly human-like.

**B. Type-Token Ratio (weight: 0.3 within Signal 2)**

AI models, when sampling with low temperature, tend toward slightly higher lexical diversity per token because they optimize toward likely completions across the full vocabulary. The TTR (unique words / total words) captures this. Only applied to texts with 50+ words — on shorter texts TTR approaches 1.0 regardless of source and adds noise rather than signal.

**C. Comma density (weight: 0.2 within Signal 2)**

AI prose reliably produces 1–3 commas per sentence through list enumeration and subordinate clause structures. This is the weakest signal (lowest weight) because it is easily disrupted by domain: technical writing uses many commas, poetry uses few, and neither is AI by that fact alone. It is included as a corroborating signal, not a primary one.

**What I would change for production:** TTR is too simplistic for a real deployment — it collapses when the text domain is narrow (a recipe, a legal clause) because all in-domain texts repeat the same vocabulary. A better replacement is mean Pointwise Mutual Information of adjacent word pairs, which measures whether word choices are predictable given their context.

## Confidence scoring

The two signal scores are combined as a weighted sum:

```
confidence = (0.6 × llm_score) + (0.4 × stylometric_score)
```

The LLM weight is higher because it reasons about semantics and structure rather than surface patterns.

### Example 1 — High-confidence AI (from Milestone 4 testing)

**Input text (AI polished paragraph):**
> "Artificial intelligence is transforming industries across the globe. Organizations are leveraging machine learning to optimize operations, reduce costs, and enhance customer experiences. The potential applications of AI are vast, spanning healthcare, finance, education, and beyond. As these technologies continue to evolve, stakeholders must consider the ethical implications and governance frameworks required to ensure responsible deployment."

| Signal | Score |
|--------|-------|
| LLM (llama3-8b-8192) | 0.85 |
| Stylometric | 0.749 |
| **Combined confidence** | **0.810** |

**Label:** `LIKELY_AI` — blocked from auto-publishing, banner shown.

Why high confidence: the LLM recognizes the professional hedging register and the stylometric signal sees low sentence-length variance (four sentences of 8, 14, 13, and 20 words — moderate rhythm), high TTR, and medium comma density all pointing in the same direction.

### Example 2 — Lower-confidence / human (from Milestone 4 testing)

**Input text (casual human writing):**
> "I burned the toast again. Third time this week, honestly. My roommate just stared at me and said nothing, which is fair. I don't even like toast that much, so why do I keep making it? Some kind of routine, I guess. Anyway. The coffee was good."

| Signal | Score |
|--------|-------|
| LLM (llama3-8b-8192) | 0.15 |
| Stylometric | 0.375 |
| **Combined confidence** | **0.240** |

**Label:** `HUMAN` — no banner, no block.

Why low confidence: the LLM correctly identifies the casual voice, self-interruptions ("Anyway."), and informal reasoning. The stylometric signal sees high sentence-length variance (CV ≈ 0.67) which drives variance_score to 0.33, low comma density (few commas relative to sentences), and a text too short for TTR to apply — all producing a below-threshold stylometric score.

The gap between these two examples (0.81 vs. 0.24) shows the scoring produces meaningful variation and is not collapsing outputs to a narrow range.

## Transparency label variants

Every `/submit` response includes a `label` field, a `block` flag, and an optional `banner` string. There are three possible variants:

**Variant 1 — AI_GENERATED** (confidence ≥ 0.90)

```json
{
  "label": "AI_GENERATED",
  "block": true,
  "banner": null
}
```

The content is blocked from public posting. The `block: true` flag signals the platform to prevent publishing. No banner is shown because the content does not reach the audience. The creator receives the classification response and must file an appeal via `POST /appeal` to contest it.

**Variant 2 — LIKELY_AI** (confidence 0.40–0.89)

```json
{
  "label": "LIKELY_AI",
  "block": false,
  "banner": "This content may be AI-generated. Review before trusting."
}
```

The content is allowed to publish, but the banner text is displayed to readers alongside it. The creator is not blocked, but the audience is warned. Appeal is optional.

**Variant 3 — HUMAN** (confidence < 0.40)

```json
{
  "label": "HUMAN",
  "block": false,
  "banner": null
}
```

No intervention. The content publishes normally with no label and no banner.

## Known limitations

### 1. ESL writers at elevated false-positive risk

A non-native English writer producing careful, grammatically correct prose will exhibit low burstiness (they avoid complex sentences they are unsure of), conservative vocabulary overlap with AI training corpora, and few contractions or idiomatic phrases. This is not because the writing is AI-like — it is because careful second-language writing shares surface features with AI output for structural reasons. The stylometric signal cannot distinguish between "low burstiness because AI" and "low burstiness because the author is writing in their second language and avoiding linguistic risk."

The 0.9 block threshold is set deliberately high to reduce this harm, but ESL writers remain at elevated risk of landing in the LIKELY_AI band (0.4–0.9) where the banner is shown. This is a fairness problem documented in the literature: Liang et al. (2023) found AI detectors disproportionately flag text by non-native English writers ([arxiv.org/abs/2304.02819](https://arxiv.org/abs/2304.02819)).

### 2. Short texts are unreliable — not for generic "insufficient data" reasons

The TTR sub-metric is hardcoded to return a neutral 0.5 for texts under 50 words. This is not just "needs more data" — it reflects a mathematical property of the metric: TTR approaches 1.0 as text length decreases regardless of source, because there are too few repetitions for patterns to emerge. A 20-word human caption and a 20-word AI caption will both show TTR near 0.95, providing zero discrimination. Similarly, CV requires at least 3 sentences; with 2 sentences any variance reading is driven by a single outlier. On short texts (a tweet, a caption, a heading), this system is running on the LLM signal alone with stylometrics contributing only noise.

### 3. Adversarial evasion is trivial

Anyone aware of what the detector measures can defeat the stylometric signal in under a minute: add one very short sentence to a long AI paragraph to spike the CV, or break one long sentence into two to reduce TTR. The detector is calibrated for unsophisticated copy-paste AI use (bulk content farms, students submitting unedited ChatGPT output), not for a determined adversary who knows the detection approach. This is not a fixable weakness with more data — it is a fundamental property of transparent, rule-based stylometric detection.

## Spec reflection

### Where the spec helped

The spec's `POST /appeal` endpoint definition — specifically the decision to write the appeal fields (`appeal_reasoning`, `appeal_timestamp`, `status`) back into the same audit log row rather than a separate table — kept the data model simple and auditable. A separate appeals table would have required a join to reconstruct a submission's full lifecycle. The spec's choice to update in-place meant `GET /log` could return the complete history of any submission, including its appeal outcome, without any extra queries. I followed this directly and it simplified both the implementation and the verification.

### Where implementation diverged from spec

The spec's original stylometric formula used `cv / 2.0` as the denominator for converting sentence-length variance into a score. During calibration, Sample A (clearly human casual text, CV ≈ 0.63) produced a stylometric score of 0.68 — well inside the AI-flagging zone — because `cv / 2.0` compresses the human-writing range (CV ~0.4–0.8) into the top half of the [0.5, 1.0] interval.

I changed the denominator to `min(cv, 1.0)`, which maps the full range of plausible human CV values (0–1.0) across the full [0.0, 1.0] score range. This made the signal directionally correct: CV = 0.63 → score = 0.37 (human-like). The change is documented in an inline comment in [`app.py`](app.py) explaining both the original formula and why it was changed, so a future reader can understand the deviation from spec without digging through the test history.

## AI usage

### Instance 1: Stylometric calibration debugging

**What I directed:** After implementing the stylometric formula from the spec, I ran the three test samples and found the human casual text (Sample A) scoring 0.68 — well above the expected < 0.45 threshold. I described the specific failure (input text, expected vs. actual sub-metric values) and asked the AI to diagnose which sub-metric was causing it.

**What it produced:** The AI correctly identified that the CV denominator of 2.0 was the problem, traced through the math showing that human casual writing (CV ≈ 0.6) maps to variance_score ≈ 0.70 under the original formula, and proposed changing the denominator to 1.0. It also separately identified that the 10-word TTR threshold was causing all three samples to return TTR ≈ 1.0 (no discrimination), and proposed raising the threshold to 50 words.

**What I revised:** I accepted both changes, but I required the AI to add inline comments explaining the reasoning for each deviation from spec — specifically *why* the original formula was wrong and what property of human writing the new formula is actually measuring. The AI's first version of the comments described what the code does ("denominator changed from 2.0 to 1.0"); I pushed back and required it to explain why that makes the signal directionally correct for real human writing ranges.

### Instance 2: `POST /appeal` endpoint implementation

**What I directed:** I gave the AI the spec for the appeal endpoint — accept `content_id` and `creator_reasoning`, look up the row, return 404 if not found, update `status` to `under_review`, write `appeal_reasoning` and `appeal_timestamp` into the same row. I also specified that the appeal fields should be added as migration guards in `init_db()` using try/except around `ALTER TABLE`.

**What it produced:** A working endpoint that matched the spec almost exactly, including the migration guard pattern. The AI added `row_factory = sqlite3.Row` to the connection object to make the lookup return a proper row proxy, which was a small but correct addition I had not specified.

**What I revised:** The AI initially structured the migration guards as a separate helper function `_migrate_db()`. I overrode this and asked it to inline them into `init_db()` as a loop over a tuple of DDL strings — consistent with how the earlier migration guards were already structured in the codebase. The helper function was not wrong, but it introduced an extra abstraction for what is already a simple pattern. I kept the code flat.
